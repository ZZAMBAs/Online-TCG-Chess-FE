# Git Commit

## 개요

현재 변경사항과 구현 의도를 확인한 뒤 repo 규칙에 맞는 한국어 커밋 메시지를 작성하고 실제 `git add`와 `git commit`을 수행한다. 별도 사용자 승인 게이트는 두지 않지만, 커밋 범위가 불명확하거나 무관한 변경이 섞여 있으면 커밋하지 않고 보고한다.

## 커밋 메시지 규칙

- 제목 형식은 `{접두어}: {커밋 메시지}`로 작성한다.
- 제목은 50자를 넘지 않는다.
- 커밋 메시지는 간결하게 쓰고 명사로 끝낸다.
- 여러 종류에 해당하면 변경의 핵심을 가장 잘 대표하는 접두어 1개만 선택한다.
- 50자 이하 제목에 커밋 이유를 담기 어렵다면 제목에는 요약만 쓰고 body에 `왜`, `어떻게`를 중심으로 적는다.
- body는 반드시 작성하지 않는다. 필요할 때만 작성한다.
- body를 작성할 때도 각 항목은 명사형으로 끝낸다.

## 접두어 선택

- `feat`: 기능 추가 및 기존 기능 동작 변경
- `fix`: 에러 수정
- `docs`: `md`, `txt` 등 문서 파일만 수정
- `chore`: 빌드 도구, 설정, 관리성 작업처럼 제품 기능과 직접 관련 없는 수정
- `refactor`: 기능 추가나 동작 변경이 없는 기존 기능 리팩터링
- `style`: 코드 스타일, 포맷, 공백, 세미콜론처럼 의미 변화 없는 수정
- `test`: 테스트 코드 수정

## 진행 절차

1. 현재 상태와 변경 범위를 확인한다.
   - `git status --short --branch`
   - `git diff --stat`
   - `git diff`
   - `git diff --staged`
2. untracked 파일이 있으면 파일명과 내용 일부를 확인해 커밋 범위에 포함할지 판단한다.
3. 변경사항이 하나의 목적을 가진 커밋인지 판단한다.
   - 하나의 목적이면 tracked와 untracked 변경 중 해당 범위 전체를 staging한다.
   - 이미 staged된 변경이 있더라도 unstaged/untracked 변경과 같은 목적이면 함께 포함할 수 있다.
   - 서로 무관한 변경이 섞여 있거나 사용자 작업을 임의로 포함할 위험이 있으면 staging/commit을 중단하고 분리 기준을 보고한다.
4. 변경의 대표 유형에 따라 접두어를 고른다.
5. 제목만으로 충분한지 판단한다.
   - 변경 이유와 방식이 diff만으로 명확하고 짧게 표현되면 제목만 사용한다.
   - 배경이나 의사결정이 중요하면 body를 추가한다.
6. `git add`로 커밋 대상 파일만 staging한다.
7. `git diff --staged --stat`과 필요 시 `git diff --staged`로 최종 커밋 내용을 확인한다.
8. `git commit`을 실행한다.
9. 완료 후 `git status --short --branch`로 남은 변경과 브랜치 상태를 확인해 보고한다.

## 메시지 예시

```text
feat: 문자열 병합 기능 추가

- A 클래스의 필요성에 의한 추가
- 기존 라이브러리 내 B 클래스 기능 보강
```

```text
docs: 커밋 규칙 문서 추가
```

## 실패 처리

- 50자 이하 제목으로 요약할 수 없으면 억지로 긴 제목을 만들지 말고 body를 사용한다.
- 커밋 대상이 불명확하면 커밋하지 않는다.
- 테스트 실패나 빌드 실패가 커밋 자체를 막아야 하는 요청이 아니라면, 실행하지 않은 검증은 완료 보고에 명시한다.
- `git commit`이 hook 실패로 막히면 실패한 hook, 핵심 오류, staged 상태 유지 여부를 보고한다.

## 완료 보고

완료 보고에는 다음만 간결히 포함한다.

- 생성된 커밋 해시와 제목
- 선택한 접두어의 이유
- body 작성 여부
- 남은 uncommitted 변경 여부
