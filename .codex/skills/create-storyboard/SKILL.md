# Create Storyboard

## 원칙

- 항상 먼저 `$spec-read`를 사용해 BE `master`의 `docs/spec/spec-fixed.md` 최신성을 확인한다.
- `$spec-read`가 실패하면 캐시가 있어도 열지 말고 스토리보드 작업을 중단한다.
- 요구사항 원문을 FE 저장소에 권위 문서로 복사하지 않는다.
- 소스코드, 앱 설정, BE 요구사항 문서는 수정하지 않는다.
- 최종 산출물은 `docs/design/storyboard.html` 하나로 둔다.
- 페이지별 임시 확인 파일은 `docs/design/tmp/*.html`에 둔다.
- 기존 `docs/design/storyboard.html`이 있으면 먼저 읽고, 새로 덮어쓰기보다 기존 의도와 구조를 유지하며 수정한다.

## 전체 흐름

1. `$spec-read`로 요구사항 최신성을 확인하고, 확인된 commit 또는 sha256을 작업 요약에 기록한다.
2. 요구사항에서 필요한 화면을 도출한다.
   - 페이지명
   - 목적
   - 주요 사용자 행동
   - 각 행동 이후의 결과 화면, 모달, 상태 변화, 오류, 또는 다음 페이지
   - 노출해야 할 서버 확정 상태
   - 오류, 제한, 권한 상태
3. 전체 톤앤매너를 먼저 정한다.
   - 도메인에 맞는 색상, 밀도, 컴포넌트 분위기를 제안한다.
   - 사용자에게 1회 확인을 받는다.
4. 도출한 페이지마다 `페이지 설명 -> 세부 질문 -> tmp HTML 생성 -> 브라우저 확인 -> 승인/반려 반영`을 반복한다.
5. 모든 페이지가 승인되면 tmp HTML들을 하나의 `docs/design/storyboard.html`로 통합한다.
6. 통합본을 브라우저로 확인하고 최종 승인을 받는다.
7. 최종 승인 후 `docs/design/tmp/` 안의 임시 스토리보드 파일을 삭제한다.

## 질문 규칙

- 질문은 한 번에 하나만 한다.
- 사용자의 답이 해결되기 전에는 다음 질문으로 넘어가지 않는다.
- 질문은 디자인 결과를 바꾸는 것만 한다.
- 요구사항에서 이미 알 수 있는 내용은 묻지 않는다.
- 질문마다 추천안을 먼저 제시하고 이유를 짧게 붙인다.
- 기본 질문 순서는 다음을 따른다.
  1. 정보 우선순위
  2. 레이아웃 밀도
  3. 상태와 오류 노출 방식
  4. 주요 행동 버튼 강조 방식
  5. 모바일에서 먼저 보여줄 정보
  6. 색상, 카드, 패널, 표 등 시각 세부값

## 페이지별 작업 규칙

- 페이지 작업을 시작할 때 어떤 페이지를 만들지 먼저 설명한다.
- 각 페이지는 독립적으로 확인 가능한 HTML로 `docs/design/tmp/<page-id>.html`에 만든다.
- tmp HTML에는 해당 페이지의 핵심 레이아웃, 상태, 오류, PC 화면, 모바일 화면을 포함한다.
- 사용자 화면 프레임은 실제 사용자가 보는 UI를 기준으로 만든다.
  - 프레임 안에는 실제 노출 문구, 버튼, 카드, 모달, 인라인 피드백, 오버레이처럼 구현 대상인 UI만 둔다.
  - 요구사항 설명, 구현 의도, 개발 상태명, BE/FE 조건 설명은 프레임 안에 섞지 않는다.
  - 실제 UI와 설명을 구분하기 위해 `actual-ui`, `ui-copy`, `story-note`, `dev-state` 같은 의미 있는 class를 사용할 수 있다.
- FE 협업 정보는 화면 밖에 따로 남긴다.
  - `story-note`는 페이지 고유의 구현 판단을 돕는 내용만 쓴다. 예: 카드 노출 규칙, 카드 사용 불가 표시 위치, 채팅 토글 동작.
  - `스토리보드 주석`이 무엇인지 설명하는 자기 설명 카드처럼 모든 페이지에 반복되는 안내는 tmp 페이지에 넣지 않는다.
  - 개발 상태명은 사용자 문구처럼 보이지 않게 `dev-state` 같은 별도 라벨이나 주석으로 표시한다.
- 각 페이지 스토리보드는 PC 화면과 모바일 화면을 명확히 나눠 표현한다.
  - 단순히 반응형으로 접히는 구조만 보여주지 말고, `PC layout`과 `Mobile layout`처럼 사용자가 구분해 검토할 수 있게 둔다.
  - PC와 모바일에서 정보 우선순위, 모달/시트 위치, 버튼 배치가 다르면 각각 별도로 표현한다.
- 해당 페이지에서 사용자가 할 수 있는 주요 행동이 있으면, 행동 이후 상태도 같은 페이지 tmp HTML에 표현한다.
  - 예: 제출 성공, 검증 실패, 서버 거부, 모달 표시, 하단 시트, 대기 상태, 비활성 상태, 자동 이동.
  - 행동 이후 상태도 실제 사용자에게 보이는 화면을 먼저 표현한다.
  - 조건, 서버 이벤트, FE 상태명, 구현 설명은 같은 사용자 화면 안에 섞지 말고 화면 밖 주석 또는 개발 상태명 라벨로 분리한다.
  - 행동 결과가 다른 페이지로 이동하는 것이라면 목적지 페이지를 예상 화면으로 만든다.
  - 이후 실제 목적지 페이지 스토리보드가 완성되면, 기존 예상 화면과 표현을 동기화한다.
- 사용자가 승인하면 다음 페이지로 넘어간다.
- 사용자가 반려하면 반려 이유를 기준으로 필요한 질문부터 다시 진행한다.
- 가벼운 문구, 간격, 색상 수정은 추가 질문 없이 반영할 수 있다.
- 요구사항과 충돌하는 요청은 즉시 알리고 BE 요구사항을 우선한다.

## 최종 통합 규칙

- 통합본은 여러 페이지를 하나의 HTML 문서에서 탐색 가능하게 구성한다.
- 공통 CSS 토큰을 사용하고 중복 스타일을 정리한다.
- 최소한 다음을 포함한다.
  - 스토리보드 읽는 법: 실제 사용자 화면, 조건부 사용자 노출, 설명 전용 주석, 개발 상태명의 구분
  - 화면 흐름 요약
  - 페이지별 와이어프레임
  - 주요 상태와 오류 예시
  - 데스크톱/모바일 레이아웃
  - 색상, spacing, radius 등 CSS 토큰
- 통합본 초입에 공통 구현 핸드오프 안내를 한 번 둔다.
  - 페이지별 tmp에는 공통 읽는 법을 반복하지 않는다.
  - 페이지별 tmp에는 해당 페이지 고유의 `story-note`, 행동 이후 상태, 개발 상태명만 남긴다.
- `docs/design/storyboard.html`은 브라우저에서 파일로 직접 열 수 있어야 한다.
- 외부 CDN, 빌드 도구, 앱 런타임에 의존하지 않는다.

## 브라우저 확인

- 정적 HTML이면 개발 서버를 띄우지 말고 파일 경로를 안내한다.
- 가능하면 headless Chrome 또는 Playwright로 데스크톱과 모바일 스크린샷을 생성해 깨짐을 확인한다.
- GUI 브라우저를 직접 여는 명령은 권한 승인이 필요할 수 있으므로, 실패하면 사용자에게 파일 경로와 실패 이유를 알려준다.
- 텍스트가 버튼, 카드, 표, 패널 밖으로 넘치거나 다른 요소를 가리면 수정한다.
- 체스보드, 카드, 관리자 목록처럼 고정 형식 UI는 `aspect-ratio`, grid, min/max 크기로 레이아웃 흔들림을 줄인다.

## 금지 사항

- 앱 소스코드를 생성하거나 수정하지 않는다.
- BE 요구사항을 수정하지 않는다.
- 요구사항을 FE 저장소에 원문 복사본으로 추가하지 않는다.
- 사용자 승인 없이 페이지별 tmp 결과를 최종본에 통합하지 않는다.
- 최종 승인 전 tmp 디렉터리를 삭제하지 않는다.
